# 2026-03-18 시스템 개선 기록

## 개요
오늘 점검에서는 그래프 퍼스트 RAG 전략 자체를 바꾸기보다, 현재 전략의 약점을 보완하는 방향으로 개선점을 정리했다. 핵심은 검색은 그래프가 맡고, 질문 성립 여부 판단과 계산/적용 판정은 점점 더 결정적으로 분리하는 것이다.

## 오늘 반영한 개선 사항

### 1. 질문 유효성 검증 레이어 추가
- 규정상 함께 성립하지 않는 질문 조건을 사전에 차단하도록 검증 레이어를 도입했다.
- 대표 사례:
  - `1급 + 49호봉`
  - 연봉제 적용 대상인데 호봉 본봉표로 직접 계산하려는 질문
- 기대 효과:
  - 잘못된 계산 방지
  - 백엔드 간 응답 불일치 감소
  - "정보 부족"이 아니라 "질문 조건 불일치"를 더 정확하게 설명 가능

### 2. Trace 패널 강화
- Trace 패널에 다음 정보를 노출하도록 개선했다.
  - 조회 언어
  - 질문 검증 결과
  - 추출 엔티티
  - 규정 컨텍스트
  - 그래프/문서 컨텍스트
- 기대 효과:
  - 디버깅 속도 향상
  - 사용자 신뢰성 향상
  - TypeDB/Neo4j/Context RAG 차이를 눈으로 확인 가능

### 3. 조회 언어 명시
- 각 경로의 trace에 다음 표기를 추가했다.
  - TypeDB KG RAG: `TypeQL`
  - Neo4j Graph RAG: `Cypher`
  - Context RAG: `Context-only`
- 기대 효과:
  - 현재 백엔드가 무엇으로 근거를 찾는지 혼동 방지
  - TypeDB가 Cypher로 조회하는 것처럼 보이는 오해 해소

## 오늘 논의로 도출한 구조적 개선 포인트

### 1. 현재 전략의 명확한 정의
현재 채택 전략은 **그래프 퍼스트 RAG**이다.

구성:
1. 엔티티 추출
2. 관련 규정 검색
3. 그래프 서브그래프 조회
4. 최종 자연어 답변 생성

보강된 요소:
- adaptive hop retrieval
- 질문 유효성 검증
- trace 기반 설명성 강화

### 2. 현재 전략의 한계
- 규정 적용 가능 여부를 완전히 결정적으로 판정하지 못함
- 계산 질문에서 LLM이 중간 추론을 많이 담당함
- 부칙/예외/우선순위 적용을 코드 레벨에서 더 명확히 실행할 필요가 있음
- TypeDB/Neo4j 간 검색은 분리되어 있지만, 실행 결과의 규정 적용 방식은 아직 충분히 명시적이지 않음

### 3. 다음 단계 권장 전략
그래프 퍼스트 RAG를 유지하되, 다음을 추가하는 하이브리드 전략을 권장한다.

**Graph-first RAG + 규칙 검증 + 결정적 계산**

의미:
- 검색: 그래프가 담당
- 적용 가능 여부 판단: 규칙 엔진/검증 레이어가 담당
- 숫자 계산: 코드가 담당
- 최종 설명: LLM이 담당

## 다음 단계 아키텍처 제안

### 1. Query IR 도입
질문을 구조화된 내부 표현으로 변환한다.
예시 필드:
- intent
- grade
- position
- eval
- step_no
- ask_type
- requires_calculation
- requires_applicability

### 2. Applicability Engine 확장
현재 질문 검증 레이어를 다음 수준까지 확장한다.
- 직급-호봉 조합 가능 여부
- 직위-직급 조합 가능 여부
- 연봉제/호봉제 적용 가능 여부
- 임금피크제 적용 가능 여부
- 부칙/예외 조항 우선 적용 여부

### 3. Retrieval Planner 도입
질문마다 필요한 근거를 먼저 계획한다.
예시:
- 조문만 조회
- 차등액표 + 상한액표 조회
- 직책급표 + 조문 + 부칙 조회

### 4. Deterministic Executor 추가
검색 결과로부터 실제 계산과 적용을 코드로 수행한다.
우선 적용 후보:
- 연봉제본봉 조정
- 연봉상한 적용
- 임금피크제 지급률 적용
- 상여금 지급률/지급시기 계산

### 5. LLM 역할 축소
LLM은 다음 역할에 집중시킨다.
- 계산 결과 설명
- 근거 요약
- 사람이 읽기 좋은 답변 구성

LLM이 직접 맡지 않게 할 것:
- 적용 대상 판정
- 상한 적용 판단
- 표 기반 숫자 계산
- 규정 충돌 해소

## 우선순위 높은 개선 후보
1. Query IR 공통 모듈화
2. 질문 검증 레이어를 규정 적용 엔진 수준까지 확장
3. 계산형 질의를 결정적 Executor로 이동
4. trace에 retrieval plan, execution steps 추가
5. TypeDB/Neo4j 공통 추출 로직 리팩터링

## 메모
현재 방향은 검색 전략 교체가 아니라 **검색 이후의 실행 레이어 강화**가 핵심이다. 즉, 그래프 퍼스트 RAG는 유지하되 규정 QA/계산 QA에 맞게 점진적으로 하이브리드 추론 구조로 고도화하는 것이 적절하다.
